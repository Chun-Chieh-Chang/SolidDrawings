# 🧠 Karpathy 資深編程準則 (Senior Coding Standards)

本文件整合了由 Andrej Karpathy 觀察到的 LLM 編程缺陷及對應的解決方案。這些準則已深度整合至 Antigravity 的核心編程邏輯中。

## 1. 先思考再行動 (Think Before Coding)
**核心：拒絕盲目假設，明確揭示權衡。**

- **設計硬門檻 (Hard Gate Design)**：在執行任何實作、寫程式碼或腳手架前，必須先呈現設計方案並獲得用戶核准。這適用於任何專案，無論多簡單。
- **蘇格拉底式探索 (Socratic Discovery)**：一次僅問一個問題，深入挖掘意圖、約束與成功標準。禁止在未釐清邏輯前跳入實作。
- **顯性假設**：在執行前，明確列出當前對需求的理解與假設。如有不確定，停止並詢問。
- **方案對比**：提供 2-3 種不同取徑及其優劣，並給予專業推薦。

## 2. 極簡主義 (Simplicity First)
**核心：用最少的代碼解決問題，絕不預測未來。**

- **YAGNI (You Aren't Gonna Need It)**：只做當前要求的，移除任何不必要的預期功能。
- **反抽象化**：對於單次使用的邏輯，嚴禁過度封裝。
- **50 vs 200 檢視**：如果 50 行就能寫完，絕不產出 200 行的代碼。

## 3. 精確微創修改 (Surgical Changes)
**核心：只動該動的地方，只清理自己造成的垃圾。**

- **精確定位**：在 `writing-plans` 中必須標註精確的檔案路徑與行數範圍。
- **孤兒清理**：僅刪除因本次修改而失效的 import、變數或函數。

## 4. 目標驅動與驗證 (Goal-Driven Execution)
**核心：定義成功標準，循環直到通過驗證。**

- **顆粒度規劃 (Bite-Sized Planning)**：將任務拆解為 2-5 分鐘的微型動作。每個動作必須包含：精確路徑、完整代碼片段、驗證命令。
- **零佔位符 (Zero Placeholders)**：計畫中嚴禁出現 "TODO", "TBD", "稍後實作" 等字眼。每一任務必須包含工程師執行所需的全部資訊。
- **驗證先行**：
  - 「增加驗證」 → 「先寫失敗測試，再讓它通過」。
  - 「修復 Bug」 → 「先寫重現測試，再修復它」。

## 5. 拒絕 Vibe Coding (Anti-Vibe Coding)
**核心：禁止盲目生成，將 AI 視為具備工程紀律的隊友。**

- **系統化除錯 (Systematic Debugging)**：嚴禁在未完成「根因調查 (Phase 1)」的情況下提出修復方案。
- **三修法則 (3-Fix Rule)**：若嘗試 3 次修復仍失敗，必須停止修補，重新審視系統架構 (Architecture Review)，並與用戶討論根本性重構。
- **垂直切片 (Vertical Slices)**：確保每一個小切片都能獨立運行與測試，拒絕一次性生成龐大架構。


---

## 📈 效益指標
當你發現以下情況時，說明這些準則正在生效：
1. **Diff 變乾淨了**：沒有不相關的代碼變動。
2. **一次就寫對**：代碼簡單易讀，不再需要反覆修改。
3. **對話更有深度**：AI 在動手前會先問出關鍵問題。
