# Gap Planner Sub-Agent

## Purpose

Transform `gap-analysis-report.md` into an actionable, MECE-compliant `task_plan.md` for the next development sprint. This skill ensures that implementation work is always driven by objective gap data, not intuition.

## Input / Output

| | Path | Format |
|:---|:---|:---|
| **Input** | `gap-analysis-report.md` | Markdown with score table, priority sections, and per-domain gaps |
| **Output** | `task_plan.md` | Markdown with phases, tasks, status tracking, and error log |
| **Context** | `progress.md` | Updated with plan summary |
| **Reference** | `PLAN.md` | Overall project plan for consistency |

## Flow

### Phase 1: Read & Parse Gap Report

1. Read `gap-analysis-report.md` fully.
2. Extract the **評分總覽** (score table) — record each domain's current score.
3. Extract **優先級行動建議** (Section 11) — P0, P1, P2, P3 items.
4. Extract **架構風險** (Section 12) — tech debt items.
5. Check each domain's detailed gap tables (Sections 1-8) — identify ❌ missing items.

### Phase 2: Prioritize & Batch

Apply these rules in order:

1. **P0 items always go first** — critical path blockers.
2. **P1 items next** — high-value features for the current sprint.
3. **Batch by domain** — group gaps from the same domain into one task (e.g., "Implement Boundary Surface + Trim Surface" as one Surfacing task).
4. **Keep sprint size 5-7 tasks maximum** — any more and quality degrades.
5. **Architecture/tech debt tasks (Section 12)** — interleave one per sprint alongside feature work.
6. **Skip domains marked "未開始" if user has postponed them** (check DEV_LOG.md for user directives).
7. **Order tasks by dependency** — backend API before frontend UI, foundation before features.

### Phase 3: Write task_plan.md

**CRITICAL RULE — Acceptance Criteria MUST be defined upfront during planning, BEFORE any code is written.**

Each task MUST include explicit, verifiable acceptance criteria written at the highest standard. These criteria serve as the contract that determines whether the task is complete — not subjective judgment, not "works on my machine." Every acceptance criterion must be:

1. **Verifiable** — can be checked by a command, a test, a UI assertion, or a diff inspection
2. **Unambiguous** — one objective interpretation only
3. **Highest standard** — matches SOLIDWORKS 2010 professional quality, not "minimal viable"

Use the existing `task_plan.md` format as template:

```markdown
# Task Plan: Sprint YYYY-MM-DD — [Domain Focus]

## Status: Planned

## Goal
[2-3 sentence sprint goal explaining what gaps this sprint closes]

---

## Sprint Tasks

### Task N: [Task Title] — [P0/P1]

**Acceptance Criteria:**
- [ ] [Verifiable criterion 1 — e.g., "POST /boundary_surface returns 200 with shape_hash when given 4 boundary curves"]
- [ ] [Verifiable criterion 2 — e.g., "tsc --noEmit shows zero new errors after changes"]
- [ ] [Verifiable criterion 3 — e.g., "UI button appears in the SURFACING tab and calls the API"]
- [ ] [Any additional highest-standard checks required]

- [ ] Subtask A: [specific action with file path]
  - [ ] Subtask A.1
  - [ ] Subtask A.2
- [ ] Subtask B

### Task N+1: [Task Title] — [P0/P1]

**Acceptance Criteria:**
- [ ] [criterion 1]
- [ ] [criterion 2]
...

---

## Gap Scoring After This Sprint (Projected)

| Domain | Current | Target |
|:---|---:|---:|
| Surfacing | 30% | 45% |
| Assembly | 25% | 35% |
...

## Errors Encountered
| Error | Attempt | Resolution |
|---|---|---|
| (none yet) | | |
```

**Rule: No task may be written without acceptance criteria.** If you cannot define verifiable acceptance criteria for a task, the task is not well-defined enough to plan — break it down further or gather more information before planning.

### Phase 4: Update progress.md

Append a new entry to `progress.md`:

```markdown
## YYYY-MM-DD — Sprint Plan: [focus]

Generated by gap-planner from gap-analysis-report.md.

**Sprint scope:** N tasks targeting [domain]
**Target score improvements:**
- Domain A: X% → Y%
- Domain B: X% → Y%

**Tasks:**
1. [task title]
2. [task title]
...
```

### Phase 5: Self-Verification

Before completing, verify:
- [ ] task_plan.md has 5-7 tasks (no more, no less than 3)
- [ ] Every task has a clear P0/P1/P2 label
- [ ] **Every task has explicit, verifiable acceptance criteria defined upfront — NOT just a verification step, but specific functional criteria**
- [ ] **Every acceptance criterion is unambiguous and objectively checkable (command output, API response, UI assertion, diff inspection)**
- [ ] **Acceptance criteria are set at the highest standard — matching SOLIDWORKS 2010 professional quality, not "minimal viable"**
- [ ] No tasks duplicate each other (MECE)
- [ ] Every task references specific files or APIs
- [ ] Every task includes a verification step
- [ ] No tasks in domains user explicitly postponed
- [ ] gap-analysis-report.md was NOT modified (read-only operation)

## Anti-Patterns

- **Don't over-plan**: 5-7 tasks per sprint. If you find yourself listing 15+ items, you're planning a roadmap, not a sprint. Split into multiple sprint plans.
- **Don't re-prioritize P0 items**: P0 means "blocker" — always include them. Only skip P0 if explicitly blocked by external dependency.
- **Don't include "investigation" tasks**: Every task must produce committed code. If investigation is needed, pair it with implementation.
- **Don't modify gap-analysis-report.md**: The planner is read-only on the gap report. Only the executor updates scores.
- **Don't ignore architecture risk**: Always include at least one tech debt task per sprint (refactoring, test infrastructure, etc.).
- **Don't define tasks without acceptance criteria**: Every task MUST have explicit, verifiable acceptance criteria written during planning. A task without acceptance criteria is not ready for planning — it's a vague wish. If you can't define verifiable acceptance criteria, you don't understand the task well enough to include it in the sprint.
- **Don't write acceptance criteria that are subjective or unverifiable**: "Works well", "looks good", "feels right" are not acceptance criteria. Every criterion must be objectively checkable: an API returns 200, a TypeScript build produces zero new errors, a UI button renders and is clickable, a specific error message appears for invalid input.
- **Don't use "verification step" as a replacement for acceptance criteria**: A verification step like "Run tsc --noEmit" is a build gate, not an acceptance criterion. Acceptance criteria define WHAT the task must achieve functionally. Both are required but they serve different purposes.
